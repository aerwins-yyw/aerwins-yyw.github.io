Behind the Projects

How I Build a Machine Learning Project

A walkthrough of the process behind the Data Science & Applied Machine Learning projects: problem framing, data, features, evaluation, and where generative AI fits in, illustrated throughout with examples from this site.

Problem Framing

Regression, Classification & Causal Inference

Every project starts with naming the actual question being asked, since that choice decides everything downstream: which models are candidates, which metrics matter, and what counts as a fair test. Most of the work on this site falls into three families.

Regression

Predicting a continuous number: the question is "how much," not "which category."

The Movie Recommendation Engine predicts a numeric star rating from 0.5 to 5, scored by how far the prediction lands from the true rating (RMSE). →

Classification

Predicting a discrete category, or the probability of belonging to one: the question is "which one."

Red Wine Quality (good or bad), SpaceX Landing (lands or doesn't), and Election Electability (win or lose) are all classification problems in this sense.

Red Wine Quality Classification → SpaceX Landing Prediction → Election Electability Prediction →

Causal Inference

A different question again: not "what will happen" but "what would have happened without the intervention." It needs a counterfactual, not just a predictive model.

Smart City Policy Impact Modeling takes this approach, building a synthetic "untreated" version of each city to compare against what actually happened. →

Data

Data Collection & Wrangling

A model is only as good as what it's trained on, and real data is rarely clean or complete. Sourcing and preparing it took a different shape in each project.

Some of it came from public APIs and scraping. The SpaceX Landing Prediction project pulled launch records from the SpaceX REST API and filled gaps with targeted web scraping, since no single source covered every field needed. Other projects started from an established benchmark dataset instead: the Movie Recommendation Engine's MovieLens 10M-rating dataset and Red Wine Quality Classification's UCI/Kaggle Portuguese wine data are both already structured, but still needed cleaning, encoding, and a proper split before they were usable.

Smart City Policy Impact Modeling needed a different shape of data altogether, consistent panel data covering the same variables for the same 500-plus municipalities across multiple years, so each city could be tracked over time rather than compared just once. And sometimes the gap isn't in how the data is structured but in how much of it exists at all: the Election Electability model covers a scale of simultaneous regional races with no close historical precedent, which is where generative AI comes in, covered further down.

Modeling

Feature Engineering

Raw fields rarely predict well on their own. Turning them into signal the model can use was often the highest-leverage step.

For the Election Electability model, that meant building features out of candidate location, party coalition support, and local demographic indicators, since a raw candidate record on its own says nothing about the regional political dynamics it's actually running inside. SpaceX Landing Prediction leaned on geospatial features drawn from launch site locations, plus statistics pulled with SQL covering launch history and payload characteristics, the kind of context that isn't sitting in a single API field but has to be derived.

Not every project needed that kind of manual construction. Red Wine Quality Classification could use its physicochemical lab measurements (acidity, sulphates, alcohol, and the rest) directly as features, so the engineering effort went into handling class imbalance instead. The Movie Recommendation Engine sits in between: movie-level and user-level bias terms were engineered by hand first, then matrix factorization took over and learned the remaining latent taste patterns on its own, rather than requiring them to be hand-designed.

Evaluation

Train, Test & Validation Splits

A model that only does well on the data it was trained on hasn't proven anything. Every project held out data it never trained on and used it exactly once to report a final number, rather than tuning against it.

How much gets held out, and how many times, depends on the project. The Movie Recommendation Engine follows the split its Harvard capstone course requires: the 10M-rating dataset is divided 90/10 into a working set and a final validation set, and only the validation set, left untouched until the very last step, produces the reported RMSE of 0.80. Everything in between, moving from the baseline model to bias terms to regularization to matrix factorization, was checked against a separate portion of the working set, so each improvement could be trusted as real rather than a lucky fit to noise.

Red Wine Quality Classification evaluated its imbalanced dataset and a balanced version separately, since a model can post a high overall accuracy while quietly failing to catch the minority "bad" class it's actually meant to flag. Where the goal is a range rather than a single number, bootstrapping fills that role: resampling the data with replacement, refitting the estimate many times over, and reading the spread of results as a confidence interval instead of one lone figure. That's how Smart City Policy Impact Modeling arrives at its 95% confidence interval around the estimated effect, alongside placebo tests that rerun the same method on municipalities that were never part of the network, as a check that the effect isn't just an artifact of the method itself.

Results

Reading the Metrics

The right metric depends on the problem type covered in the first section: a regression error metric doesn't tell you what a classification metric tells you, and neither tells you what a causal-inference project needs.

RMSE (Root Mean Squared Error)

For regression: the average size of a prediction's error, in the same units as the target, with larger errors penalized more heavily. Movie Recommendation Engine: RMSE 0.80 on a 0.5–5 star scale.

AUC (Area Under the ROC Curve)

For classification: how well a classifier ranks positive cases above negative ones across all possible thresholds, not just one. Red Wine Quality Classification: AUC 0.880 (balanced KNN), 0.858 (imbalanced Logistic Regression).

Sensitivity / Specificity

For classification: the trade-off between catching true positives and avoiding false positives, used to pick a deployment threshold instead of defaulting to 0.5. Applied in Red Wine Quality Classification to choose where to draw the good/bad line.

Effect Size vs. Synthetic Counterfactual

For causal inference: the gap between what actually happened and what the model estimates would have happened without the policy. Used in Smart City Policy Impact Modeling instead of a predictive-accuracy metric, since the goal is attribution, not prediction.

AI-Assisted Workflow

Where Generative AI Fits In

Generative AI shows up here in a specific role: closing a data gap, not replacing the modeling process. The standard term for this is data augmentation, generating additional synthetic training examples so a model has enough signal to learn from. The Election Electability project needed exactly that. Indonesia's 2024 simultaneous regional elections had no comparable historical precedent at that scale, so there wasn't enough labeled history to train on directly, and generative AI was used to synthetically augment the historical training data, giving the already feature-engineered model enough examples to work with.

It's also part of the iterative loop itself, not just the data behind it. Drafting hypotheses to test during exploratory data analysis, suggesting features worth engineering, sanity-checking code before a run, and helping structure write-ups all shorten how long it takes to try an idea and then keep or discard it. What it doesn't replace is judgment: which features actually make sense, which metric fits the problem, and how to read a result once the model runs are decided and checked by hand.
