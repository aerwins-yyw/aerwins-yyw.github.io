Behind the Projects

How I Build a Machine Learning Project

A walkthrough of the process behind the Data Science & Applied Machine Learning projects — problem framing, data, features, evaluation, and where generative AI fits in — illustrated with examples from this site.

Problem Framing

Regression, Classification & Causal Inference

Every project starts with naming the actual question being asked, since that choice decides everything downstream: which models are candidates, which metrics matter, and what counts as a fair test. Most of the work on this site falls into three families.

Regression

Predicting a continuous number — the question is "how much," not "which category."

Example: Movie Recommendation Engine — predicts a numeric star rating (0.5–5), scored by RMSE. →

Classification

Predicting a discrete category, or the probability of belonging to one — the question is "which one."

Example: Red Wine Quality (good/bad), SpaceX Landing (lands/doesn't), and Election Electability (win/lose) are all classification problems.

Causal Inference

A different question again: not "what will happen" but "what would have happened without the intervention." It needs a counterfactual, not just a predictive model.

Example: Smart City Policy Impact — builds a synthetic "untreated" version of each city to compare against. →

Data

Data Collection & Wrangling

A model is only as good as what it's trained on, and real data is rarely clean or complete. Sourcing and preparing it took a different shape in each project:

  • Public APIs & scraping — SpaceX Landing Prediction pulled launch records from the SpaceX REST API and filled gaps with targeted web scraping.
  • Established benchmark datasets — the Movie Recommendation Engine used the MovieLens 10M dataset, and Red Wine Quality Classification used the UCI/Kaggle Portuguese wine dataset; both already structured but still needing cleaning, encoding, and a train/test split.
  • Panel data — Smart City Policy Impact Modeling required consistent multi-year, multi-municipality panel data (500+ municipalities) so each unit could be tracked over time, not just compared once.
  • Scarce historical data — the Election Electability model covers a scale of simultaneous regional races with no close historical precedent; how that gap was closed is covered below, under Generative AI.

Modeling

Feature Engineering

Raw fields rarely predict well on their own. Turning them into signal the model can use was often the highest-leverage step:

  • Election Electability — candidate location, party coalition support, and local demographic indicators were engineered into features, since raw candidate records alone don't capture regional political dynamics.
  • SpaceX Landing Prediction — geospatial features from launch site locations and SQL-derived statistics (launch history, payload characteristics) fed the classifiers.
  • Red Wine Quality Classification — used the physicochemical lab measurements directly (acidity, sulphates, alcohol, etc.) as features, with the engineering effort spent on handling class imbalance instead.
  • Movie Recommendation Engine — engineered movie-level and user-level bias terms before moving to matrix factorization, which learns latent taste features automatically rather than by hand.

Evaluation

Train, Test & Validation Splits

A model that only does well on the data it was trained on hasn't proven anything. Every project held out data it never trained on, and used it exactly once to report a final number.

In the Movie Recommendation Engine, that meant validating each modeling step — baseline, then bias terms, then regularization, then matrix factorization — against a held-out test set at every stage, not just at the end, so improvements were real and not overfitting to noise. In Red Wine Quality Classification, the imbalanced dataset and a balanced (resampled) version were evaluated separately, since a model can look strong on accuracy alone while quietly failing the minority class.

Results

Reading the Metrics

The right metric depends on the problem type from the first section — a regression error metric doesn't tell you what a classification metric tells you, and neither tells you what a causal-inference project needs.

RMSE — Root Mean Squared Error

For regression: the average size of a prediction's error, in the same units as the target, with larger errors penalized more heavily. Movie Recommendation Engine: RMSE 0.80 on a 0.5–5 star scale.

AUC — Area Under the ROC Curve

For classification: how well a classifier ranks positive cases above negative ones across all possible thresholds, not just one. Red Wine Quality Classification: AUC 0.880 (balanced KNN), 0.858 (imbalanced Logistic Regression).

Sensitivity / Specificity

For classification: the trade-off between catching true positives and avoiding false positives, used to pick a deployment threshold instead of defaulting to 0.5. Applied in Red Wine Quality Classification to choose where to draw the good/bad line.

Effect Size vs. Synthetic Counterfactual

For causal inference: the gap between what actually happened and what the model estimates would have happened without the policy. Used in Smart City Policy Impact Modeling instead of a predictive-accuracy metric, since the goal is attribution, not prediction.

AI-Assisted Workflow

Where Generative AI Fits In

Generative AI shows up in this work as a tool for a specific, named gap, not a replacement for the modeling process. In the Election Electability project, Indonesia's 2024 simultaneous regional elections had no close historical precedent at that scale, so there wasn't enough labeled history to train on directly; generative AI was used to synthetically expand the historical training data, so the feature-engineered model had enough examples to learn from.

More generally, it's also part of the day-to-day workflow across these projects: drafting and sanity-checking exploratory data analysis, reviewing code before a run, and helping structure write-ups — with the modeling decisions, feature choices, and metric interpretation done and verified by hand.
